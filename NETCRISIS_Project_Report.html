1. Introduction

NETCRISIS is a real-time network crisis simulation platform that combines classical computer networking concepts with modern AI agents. It models a 16-node enterprise network topology using NetworkX and visualizes it through a cinematic dark-themed D3.js dashboard. Four autonomous AI agents — powered by Groq's LLaMA 3.3 70B large language model — continuously attack, defend, monitor, and generate traffic across the network.

The system demonstrates several core computer networking concepts in action: OSPF routing with Dijkstra's algorithm, BGP route tables, access control lists (ACLs), link utilization and congestion, DDoS flood attacks, BGP hijacking, man-in-the-middle interception, and port scanning. All of these run on a real Python backend with a NetworkX graph, not simulated in the browser.

2. Project Objectives

  • Model a realistic enterprise network topology with routers, switches, servers, and hosts
  • Implement OSPF shortest-path routing using Dijkstra's algorithm via NetworkX
  • Maintain BGP routing information bases for all core and edge routers
  • Implement an ACL (Access Control List) engine for per-node traffic filtering
  • Simulate four types of cyber attacks with progressive, realistic impact
  • Build four autonomous AI agents that make strategic decisions about the network
  • Orchestrate agents using LangGraph's StateGraph with controlled firing frequency
  • Stream real-time state updates via WebSocket to a cinematic web dashboard
  • Provide manual control through REST API endpoints for human intervention

Data Flow

  1. The simulation engine runs a tick loop (~1 tick/second) updating all node health, link utilization, and active attacks
  2. Every 4 ticks, the LangGraph orchestrator invokes all four AI agents sequentially
  3. Agent decisions (attack, defend, monitor, generate traffic) are applied to the NetworkX graph
  4. The complete state is serialized to JSON and broadcast via WebSocket to all connected frontend clients
  5. The frontend receives the state and updates D3.js topology, health dashboard, logs, and top bar
  6. User actions (launch attack, cut link, reset) are sent as REST API calls to the backend

4.2 OSPF Routing (Dijkstra's Algorithm)

The simulation implements OSPF (Open Shortest Path First) routing using Dijkstra's shortest path algorithm provided by NetworkX. Each link has a dynamic cost calculated as: 

cost = 1 + (link_utilization / 100)

This means heavily utilized links have higher costs, causing traffic to prefer less congested paths — exactly how OSPF with load-sensitive metrics works in real networks. Severed links are excluded from the routing graph, and isolated nodes are unreachable.

4.3 BGP Route Tables

Every core and edge router maintains a BGP Routing Information Base (RIB). Each entry contains:

  • Destination IP — the target node's IP address
  • Next Hop — the immediate neighbor to forward to
  • AS Path — the full route path (as node IDs)
  • Metric — hop count to destination
  • Origin — route origin type (IGP for internal routes)

BGP tables are recomputed every 10 ticks and whenever a link is severed or restored. A BGP hijack attack can inject false routes into these tables.

4.4 Access Control Lists (ACLs)

Each node has an ACL — a list of rules that filter incoming traffic. Rules are structured as {action: "deny"|"permit", src: "ip"|"any"}. The defender agent can dynamically add ACL rules to block malicious traffic sources.

5. Simulation Engine

The simulation engine (simulation.py) is the core of the backend. It manages all network state using a NetworkX undirected graph where nodes are network devices and edges are links.

5.1 Tick Loop

Each tick executes these steps in order:

  1. Update Baseline Traffic — Random walk on link utilization (±3%, bounded 5–35%)
  2. Process Active Attacks — Apply damage, increase congestion, generate malicious packets
  3. Auto-Recovery — Heal compromised nodes (+1.5 hp/tick), recovering nodes (+2), normal nodes (+0.5)
  4. Generate Normal Traffic — 8 predefined traffic patterns with probabilistic activation
  5. Health Calculation — Weighted average: Core ×3, Server ×2.5, Edge ×2, Host ×1
  6. Crisis Detection — If health drops below 30%, state changes to CRISIS
  7. BGP Rebuild — Every 10 ticks, recompute all BGP routing tables

5.2 Node State Machine

  normal ──(attack)──► under_attack ──(health≤20)──► compromised
    ▲                       │                            │
    │                  (attack ends)               (auto-heal)
    │                       ▼                            ▼
    └───(health≥90)── recovering ◄──(health>50)──────────┘

  Any state ──(isolate action)──► isolated

5.3 Health Calculation

Overall network health is a weighted average of all node health values:

Health = Σ(node.health × weight[node.type]) / Σ(weight[node.type])

Weights: core=3, server=2.5, edge=2, host=1

This ensures core infrastructure impacts the overall score more heavily than endpoint hosts.

6. AI Agent System

The agent system (agents.py) implements four LangChain chains using Groq's LLaMA 3.3 70B Versatile model. Each agent has a distinct persona, system prompt, and structured output schema.

6.1 REDSTORM — Attacker Agent

Personality: Calculated, aggressive, methodical. Seeks vulnerabilities and exploits them ruthlessly.

Capabilities: Launches DDoS, BGP Hijack, MITM, or Port Scan attacks. Selects targets based on strategic value — prefers servers and core routers. Adapts strategy based on network health and active defenses.

Output: {should_attack, attack_type, target, reasoning}

6.2 GUARDIAN — Defender Agent

Personality: Calm, strategic, protective. Thinks two steps ahead of the attacker.

Capabilities: Rate-limits DDoS (30% reduction), isolates compromised nodes, adds ACL rules, applies emergency patches (+15 HP), restores severed links.

Output: {should_act, action, target, reasoning}

6.3 OVERWATCH — Monitor Agent

Personality: Analytical, precise, vigilant. Nothing escapes detection.

Capabilities: Generates alert messages, rates severity (normal/elevated/critical), tracks health trends, detects anomalies in utilization and node status.

Output: {alerts[], severity, summary}

6.4 FLOWMASTER — Traffic Agent

Personality: Steady, adaptive, realistic. Simulates real user behavior.

Capabilities: Generates 2–5 traffic flows per cycle between hosts and servers. Adapts to outages — avoids down servers and congested paths.

Output: {flows[{src, dst}], reasoning}

6.5 Fallback Logic

When the Groq API key is not configured, each agent falls back to rule-based decision making. The attacker uses randomized target selection, the defender checks for under_attack/compromised states, the monitor parses the state text for anomalies, and traffic generates random host-to-server flows.

7. LangGraph Orchestration

The agent orchestration (graph.py) uses LangGraph's StateGraph to coordinate all four agents in a structured pipeline.

7.1 State Definition

class AgentState(TypedDict):
    tick: int                           # Current simulation tick
    health: int                         # Overall network health
    network_summary: str                # Condensed state for LLM context
    active_attacks: List[Dict]          # Currently active attacks
    attacker_decision: Dict             # Output from attacker
    defender_decision: Dict             # Output from defender
    monitor_report: Dict                # Output from monitor
    traffic_plan: Dict                  # Output from traffic
    actions_to_apply: List[Dict]        # Accumulated actions (reducer: append)
    agent_logs: List[Dict]              # Accumulated logs (reducer: append)

7.2 Execution Flow

    START
      │
      ▼
  [ATTACKER] ──► Decides whether to attack, which type, which target
      │
      ▼
  [DEFENDER] ──► Sees attacker's action + network state, decides counter-move
      │
      ▼
  [MONITOR]  ──► Observes everything, generates alerts and severity rating
      │
      ▼
  [TRAFFIC]  ──► Adapts traffic patterns to current network conditions
      │
      ▼
     END ──► actions_to_apply[] and agent_logs[] are returned

7.3 Firing Frequency

Agents fire every 4 ticks (configurable via AGENT_FREQUENCY). Between agent cycles, the simulation runs purely on its tick engine — processing existing attacks, healing nodes, and generating baseline traffic. This prevents LLM latency from slowing down the simulation.

12. Setup & Execution

Prerequisites

  • Python 3.10 or higher
  • Node.js (for static file server) or any HTTP server
  • Groq API key (free tier available at https://console.groq.com/keys)

Step 1: Install Backend Dependencies

cd backend
pip install -r requirements.txt

Step 2: Configure Environment

copy .env.example .env
# Edit .env and set: GROQ_API_KEY=gsk_your_actual_key_here

Step 3: Start Backend Server

python main.py
# Server starts at http://localhost:8000

Step 4: Serve Frontend

cd ..     # back to project root
npx -y serve .
# Frontend serves at http://localhost:3000

Step 5: Open Dashboard

Navigate to http://localhost:3000 in your browser. The dashboard auto-connects to the backend and the simulation starts automatically.

13. Conclusion

NETCRISIS successfully demonstrates the intersection of classical computer networking and modern AI. By modeling a real network topology with OSPF routing, BGP tables, and ACL filtering in NetworkX, and then layering four autonomous LLM-powered agents on top, the project creates a dynamic, realistic simulation of network operations during a cyber crisis.

The system showcases how AI agents can make strategic decisions about network security — the attacker probes weaknesses, the defender responds adaptively, the monitor maintains situational awareness, and the traffic agent simulates realistic user behavior. All of this is visualized through a cinematic real-time dashboard that provides immediate feedback on every network event.

Key Computer Networking concepts demonstrated:

  • Network topology design (hierarchical core-edge-host architecture)
  • OSPF routing with Dijkstra's shortest path algorithm
  • BGP routing information bases and route table computation
  • Access Control Lists for traffic filtering
  • Link utilization, bandwidth, and congestion modeling
  • DDoS, BGP hijacking, MITM, and port scanning attacks
  • Defense mechanisms: rate limiting, isolation, ACLs, patching
  • WebSocket real-time communication protocol
  • REST API design for network management operations
